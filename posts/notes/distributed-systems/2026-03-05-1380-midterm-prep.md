Timestamps are universal points in time. 

###### clock skew
- If skew is small, less than 125 ms, we can skew the clock back to allow the clocks to sync. This will allow the clocks to sync up within 5 minutes. 

- If skew is larger, between 125 ms to 1000s, step the clock. This resets it to the estimated server timestamp. 

- If skew is even larger > 16 minutes, greater than 1000 seconds, panic and wait for human resolution. 

With corrections, clock differences can become negative.

###### Logical Time
The happens before relation $\rightarrow$ between $\alpha$ and $\beta$ is when 
- $\alpha$ and $\beta$ occur at the same node, and $\alpha$ occurs before $\beta$, or
- $\alpha$ is the sending of message m, and $\beta$ is the receipt of the same message m.

It also takes events $\gamma$ where $\alpha \rightarrow \gamma$ and $\gamma \rightarrow \beta$. 

There is potential causality when $\alpha \rightarrow \beta$, but when $\alpha || \beta$ , 
$\alpha$ could not have caused $\beta$.

**Lamport Clocks**
- Each node maintains a counter, starting at 0 and incremented on each event
- Sender attaches counter value to messages
- Receipient moves clock to max of (local, incoming), *then increments by 1*

Lamport clocks do *not* imply *happens before*
But *a happens before b* does imply L(a) < L(b)

**Vector Clocks**
- Each node maintains a vector clock = $[N_1, N_2, \ldots N]$
- Vector timestamp of an event a is $V(a) = [t_1, t_2, \ldots, t]$
- On *any event*, increment the local vector element
	- E.g. B receives a message from A which is [2,0,0]
		- Becomes [2,1,0] on receipt.
Then, on the consequent send of a message from B to C, we send 
[2,2,0], because we increment prior to send.

##### Sharding

Q: How do we know *scalably* which node stores a certain data object?
Desired properties: 
- good load balancing (LB)
- no single point of contention (no SPOC)
- no additional index space (noAddtl)
- dynamic object structure (dyn)
	- Allows update of objects via keys.
- support for node restructuring (restruct)
	- Allows adding and removing nodes with guarantee of consistency.

Idea 1: A centralized coordinator
- This is load balancing, and able to accurately round-robin across shards. It is also dynamic, because it is easy to update hashed key. It supports node restructuring because it is easy to update existing nodes.
- However, there is a SPOC, it takes up additional space, and is non-injective.

Idea 2: distribute the alphabetical function $\text{key}[0] \in a-f: n_1$, $g -m: n_2 \ldots$
- does not have a single point of contention, has dynamic object structure, and has no additional index space
- However, it does not have good load balancing, and no support for node restructuring

Idea 3: use a hash function
- Load balancing, no single point of contention, no additional index space, dynamic object structure. However, no support for node restructuring

###### total order definition
If a process delivers both m1 and m2, then all processes delivering m1 and m2 must deliver.

###### casual definition
*if m1 send before m2 send, then m1 delivery must happen before m2 delivery*.

##### not casual, FIFO example

 If you have a causal system, you have a guarantee that messages send and deliver according to the $\rightarrow$ (happens before).  For instance, if the system really was causal, from B's perspective after receiving A this would allow you to send to arbitrary third parties knowing A's receipt occurred there first. But with this system, this is not the case. "hi too" from B is received after "hi!" from A, which confuses the recipient at C.

The message sent from node B reaches C before the first message from A, so it is not causally ordered. However, it is FIFO ordered because ordering of messages from the same node are preserved.

##### cooperative vs preemptive scheduling

**cooperative scheduling:**
- Switching between tasks is typically faster, as switching happens at the appropriate time and with minimal state
- It might accidentally or purposefully lead to resource exhaustion when a task doesn't yield
- Execution up to the point a task yields should be expected to be atomic. 

> If a cooperatively scheduled task performed an action that left the system in an inconsistent state, and then yielded, that could lead to incorrect execution (similar to if a task was preempted while performing an operation without a lock).

**preemptive scheduling**.
- It is slower, partly because it might happen at inconvenient times when the task is not prepared to yield
- It is fair, because it can switch at predetermined intervals and thus give each task a fair amount of time to run.

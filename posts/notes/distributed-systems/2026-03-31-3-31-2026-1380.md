##### Read-your-writes consistency

==Writes done by a single process are seen by all processes in the order they were written by that process== .

> ###### Warning #####
> When the client issues the Read on a Replica that has yet to receive the write, it leads to an undesirable behavior wherein the client will see the old value (or null) and think that the write it made was lost.

**Read-Your-Writes consistency** states that the system guarantees that, once an item has been updated, any attempt to read the record by the same client will return the updated value. Will reassure the user their own write was successful. The ==x=?== values do not get writes which were made later on.

*Primary Backup Replication*
We might have a node `c` which writes to a node `P`, which is designated to be the primary backup node. Write the same update across different nodes: 1,2,3. These nodes could reach the primary out of order. After the ==Primary== has received all messages, the primary notifies the client that the request has succeeded.
